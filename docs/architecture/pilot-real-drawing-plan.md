# План Пилота Реального Чертежа

Этот документ — верхний контракт для ближайших задач. Он не заменяет `AGENTS.md`, `module-map.md` и `domain-concepts.md`, а связывает их вокруг одной цели: довести пилот до проверяемой цепочки от реальных данных оборудования до DXF.

## Главная Цель

Пилот должен доказать сквозной сценарий:

`паспортные данные оборудования -> EquipmentDefinition/ConnectionPoint -> Project -> EngineeringDrawing -> CadDrawing -> DXF -> evidence/status report`

Цель не в расширении UI и не в декоративной отрисовке. Цель — чтобы один простой комплект котельной строил инженерно осмысленный лист и показывал, какие данные подтверждены источниками, а какие требуют проверки инженером.

## Зафиксированный Сценарий

- Юрисдикция: РФ.
- Объект: отдельностоящий блочный/модульный блок котельной.
- Тип: газовая водогрейная котельная.
- Состав пилота: один малый котел, коллектор подачи, коллектор обратки, подача, обратка, газ, дымоход.
- Текущий котел: `rgt-100-ksva-100`.
- Текущие коллекторы: `supply-header`, `return-header`.
- Статус данных: `review_required`, если нет отдельной процедуры проверки источника и применимости.

## Definition Of Done Для Пилота

Пилот считается доведенным до первого результата, когда:

- карточки оборудования содержат реальные или явно помеченные pilot-параметры: габариты, DN, топливо, мощность, точки подключения, source document id и статус доверия;
- технологическая схема и лист A3 строятся из `EquipmentDefinition.connectionPoints` и metadata, а не из произвольных чисел внутри React;
- DXF экспортируется через `Project -> EngineeringDrawing -> CadDrawing -> AsciiDxfWriter`;
- на листе есть корпуса оборудования, патрубки, DN, линии подачи/обратки/газа/дымохода, арматура, направление потока, предупреждения о статусе данных;
- есть машинная проверка, что элементы чертежа не выходят за sheet/viewports;
- есть отчет или модель evidence, где видно: источник, статус, что подтверждено, что условно и что требует инженерной проверки;
- `npm test`, `npm run lint`, `npm run build` проходят после изменений.

## Приоритет Работ

1. Усилить данные оборудования: реальные паспорта/каталоги, source ids, extracted facts, координаты и DN патрубков.
2. Усилить `EngineeringDrawing`: символы оборудования, патрубки, подписи, слои и лист должны вытекать из модели.
3. Усилить DXF: CAD-примитивы, слои, типы линий, подписи и совместимость с CAD-просмотрщиками.
4. Добавить evidence/status report для пилота: не нормативная экспертиза, а прозрачная проверяемость данных.
5. Только после этого расширять AI-агентов, нормативные проверки и оркестрацию.

## Что Не Делать В Этом Потоке

- Не уходить в двери, окна, красивые панели и общий UX, если это не блокирует пилотный чертеж.
- Не заменять план помещения, пока третий вид `Чертёж / Предпросмотр листа` не докажет полезность.
- Не объявлять соответствие ГОСТ/СП без проверенной evidence-модели, citations и применимости.
- Не строить DXF из React SVG, DOM, screenshot или canvas.
- Не переносить файлы и не делать большой FSD-рефакторинг ради порядка.
- Не добавлять новые AI-агенты раньше, чем появится достаточная структурированная модель чертежа и evidence.

## Навигация По Существующим Файлам

- `AGENTS.md` — обязательные правила для coding agents: язык UI, слои, state, domain, safety, export architecture, тесты.
- `docs/architecture/module-map.md` — где находятся модули, которые нужно менять.
- `docs/architecture/domain-concepts.md` — смысл доменных сущностей и запрет смешивать похожие понятия.
- `src/infrastructure/equipment-catalogs/MockEquipmentCatalog.ts` — текущий pilot catalog.
- `data/evidence/typical-standalone-boiler-room/*` — текущий pilot evidence dataset.
- `src/infrastructure/drawing/BoilerRoomSheetDrawingService.ts` — сборка пилотного листа A3.
- `src/infrastructure/drawing/EngineeringDrawingToCadService.ts` — переход из `EngineeringDrawing` в `CadDrawing`.
- `src/infrastructure/exporters/AsciiDxfWriter.ts` — ASCII DXF writer.
- `src/tests/sheetDrawing.test.ts` и `src/tests/exporters.test.ts` — минимальная защита чертежа и DXF.

## Текущий Статус

На момент создания этого документа уже есть:

- pilot-котел `rgt-100-ksva-100` с `review_required` паспортными данными;
- pilot-коллекторы подачи и обратки;
- план помещения, принципиальная схема и предпросмотр листа;
- SVG листа и DXF export через drawing/CAD model;
- viewport/bounds validation для листа;
- первые DN/паспортные подписи, связанные с `ConnectionPoint` и metadata.

Главный следующий фокус: заменить условные координаты и pilot-коллекторы на более проверенные данные источников, затем сделать evidence/status report для полученного листа.
