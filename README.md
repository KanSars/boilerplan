# Boilerplan AI v0

Boilerplan AI v0 — runnable-приложение для компоновки котельной с доменной моделью в основе. UI приложения сейчас на русском языке. Проект использует Next.js, React, TypeScript и SVG-редактор плана. Вся доменная геометрия хранится в миллиметрах; перевод в пиксели выполняется только на уровне UI.

## Чем Проект Не Является

Это не сертифицированный инженерный инструмент и не замена инженеру-проектировщику котельных. Приложение не выполняет реальные гидравлические расчёты и не гарантирует соответствие ГОСТ, СП, СНиП, EN, DIN или любым другим юридическим, пожарным, газовым, вентиляционным, строительным или производственным требованиям.

## Как Пользоваться

Задайте ширину, длину и высоту помещения в метрах. Добавьте котлы и коллекторы из тестового каталога. Перетаскивайте оборудование на SVG-плане, выбирайте элементы для изменения обозначения или поворота на 90 градусов. Чтобы удалить выбранный элемент, нажмите кнопку `Удалить элемент` в панели свойств или клавишу `Delete`/`Backspace` вне текстового поля.

После компоновки можно построить предварительные трассы труб, просмотреть отчёт проверки и экспортировать JSON проекта, SVG-чертёж или CSV-ведомость оборудования. При удалении оборудования связанные трассы труб автоматически удаляются, а проверка запускается заново.

## Инженерные Представления

Редактор показывает три синхронизированные инженерные проекции одной доменной модели:

1. `План помещения` — физическая компоновка оборудования в котельной: границы помещения, корпуса, зоны обслуживания, точки подключения и предварительные физические трассы труб.
2. `Принципиальная схема` — логическая схема соединений: котлы подключаются к коллектору подачи и коллектору обратки независимо от точной геометрии трасс на плане.
3. `Чертёж / Предпросмотр листа` — print-like SVG preview листа A3 с рамкой, штампом, планом размещения, технологической схемой, DN-подписями, стрелками направления и условными символами арматуры.

Физическая `PipingRoute` и логическое `SystemConnection` — разные понятия. `PipingRoute` хранит геометрию полилинии для плана и будущих CAD/DXF экспортов. `SystemConnection` описывает инженерную связность: например, точка подачи котла подключена к коллектору подачи.

Логические соединения не хранятся в проекте как stale derived data. Они вычисляются детерминированным `SystemConnectionResolver` из текущего оборудования и точек подключения. В v0 resolver поддерживает простую схему: котёл -> коллектор подачи и котёл -> коллектор обратки. Если нужного коллектора нет, схема показывает красный крест и текст проблемы. Если подходящих коллекторов несколько, v0 выбирает первый детерминированно и сообщает предупреждение.

Лист предпросмотра строится не из DOM и не из React SVG редактора, а из отдельной `EngineeringDrawing`-модели. Поэтому SVG листа и DXF используют одну и ту же чертёжную модель.

## Точки Подключения Оборудования

Точки подключения хранятся в `EquipmentDefinition`, то есть в master data каталога, а не создаются пользователем заново в каждом проекте. При размещении оборудования на плане экземпляр автоматически получает world connection points на основе:

- `definition.connectionPoints`
- `instance.position`
- `instance.rotationDeg`

Это позволяет одной и той же доменной модели порождать сразу несколько инженерных представлений: физический план, логическую схему и предварительные физические трассы.

Архитектура готова к будущим источникам данных точек подключения:

- ручной каталог;
- AI extraction из PDF/мануалов;
- BIM/Revit/IFC import;
- user override.

## Редактор Оборудования

В левой панели есть `Редактор оборудования`. Он редактирует карточки активного каталога: название, производителя, модель, категорию, габариты, зоны обслуживания и точки подключения. Изменения сначала находятся в локальном draft-состоянии формы, затем применяются кнопкой `Сохранить`.

Карточка оборудования является master data. Размещённые элементы хранят только `definitionId`, поэтому после сохранения карточки уже размещённые экземпляры автоматически используют новые габариты, зоны обслуживания и точки подключения. Это уменьшает ручную работу в каждом проекте: пользователь редактирует карточку оборудования один раз, а план, схема, проверка и маршрутизация пересчитываются от актуального каталога.

При сохранении карточки предварительные трассы труб очищаются, потому что старая геометрия может стать неверной после изменения габаритов или точек подключения. Новые трассы строятся кнопкой `Соединить автоматически`.

## Источники Данных Оборудования

Сейчас активный каталог хранится в Redux state и инициализируется pilot/mock-данными. Первая карточка — `Котёл RGT-100 / КСВА-100`, заведённая по публичному паспорту RGT-100...500 со статусом `review_required`. Габариты, DN и параметры взяты как pilot data, а координаты патрубков остаются условными до проверки по заводскому чертежу.

В будущем тот же слой можно заменить или расширить:

- mock/test data;
- ручной каталог;
- AI extraction из PDF/мануалов;
- BIM/Revit/IFC import;
- user override.

В v0 нет базы данных, поэтому изменения каталога живут в памяти текущей сессии. JSON export включает `catalogSnapshot`, чтобы сохранить текущие карточки оборудования и точки подключения вместе с проектом.

## Автоматическое Соединение

Кнопка `Соединить автоматически` делает три вещи в одном шаге:

1. Определяет совместимые точки подключения через `ConnectionCompatibilityService`.
2. Строит логические `SystemConnection` через `SystemConnectionResolver`.
3. Генерирует предварительные физические `PipingRoute` от world connection points.

В v0 совместимость упрощена:

- подача соединяется с подачей;
- обратка соединяется с обраткой;
- газ с газом;
- drain с drain;
- signal с signal;
- electrical с electrical;
- flue с flue.

Если подходящая цель не найдена, UI показывает проблему явно. Если найдено несколько вариантов, система выбирает ближайший детерминированно и добавляет предупреждение. Ручной editor override пока не реализован и остаётся TODO на следующий этап.

## DXF Экспорт

Кнопка `Экспорт DXF` создаёт предварительный CAD-чертёж листа в ASCII DXF AutoCAD 2000 (`AC1015`). Экспорт строится не из SVG, а из доменной модели через промежуточные `EngineeringDrawing` и `CadDrawing`. Единицы листа — миллиметры бумаги.

Координаты экспортируются в CAD-ориентации: внутри preview `Y` направлен вниз, а в DXF exporter применяется преобразование `Ycad = sheet.height - Ypreview`, чтобы в CAD `Y` был направлен вверх.

Слои v1:

- `SHEET_FRAME` — рамка листа.
- `TITLE_BLOCK` — штамп и служебные поля.
- `AR_ROOM_WALL` — контур помещения.
- `ME_EQ_BODY` — корпуса оборудования.
- `ME_CONN_POINT` — точки подключения.
- `ME_PIPE_SUPPLY` — трубопровод подачи.
- `ME_PIPE_RETURN` — трубопровод обратки.
- `ME_PIPE_GAS` — газ.
- `ME_PIPE_FLUE` — дымоход.
- `ME_VALVE` — условные символы арматуры.
- `AN_TEXT` — подписи.

DXF v1 содержит `LWPOLYLINE`, `CIRCLE` и `TEXT`. Условные обозначения арматуры и DN-подписи являются pilot-графикой со статусом `review_required`; размерные линии, полноценная спецификация, блоки, hatch и DWG-конвертация оставлены на следующие этапы. Первичная проверка предполагается в nanoCAD Free или другом CAD, который читает AutoCAD 2000 DXF.

## Текущая Структура

Архитектура остаётся доменно-ориентированной и немного приближена к Feature-Sliced Design без большого рискованного рефакторинга:

- `src/app`: Next.js routes и глобальная оболочка приложения.
- `src/widgets/boiler-room-editor`: главный виджет, который собирает сценарий редактора из feature-компонентов.
- `src/features/boiler-room-editor`: UI-панели и SVG-редактор для текущего вертикального среза.
- `src/domain`: независимая доменная модель, геометрия, проверка, маршруты, экспортные интерфейсы и helper удаления оборудования.
- `src/infrastructure`: pilot/mock каталог, демо-профиль правил, роутер труб, drawing services, экспортёры и mock AI adapters.
- `src/store`: Redux Toolkit store, `projectSlice`, `editorSlice`, typed hooks и selectors для derived validation/system connections.
- `src/shared`: общая конфигурация, пока используется для shared catalog data.
- `src/lib`: общие утилиты. TODO: при следующем безопасном рефакторинге перенести в `src/shared/lib`.

Полный FSD-переезд пока намеренно не сделан: текущая задача была про исправление runnable vertical slice, а не про крупную смену структуры. Следующий безопасный шаг — разделить `features/boiler-room-editor` на отдельные slices вроде `room-settings`, `equipment-catalog`, `equipment-deletion`, `pipe-routing`, `project-export` и `validation-report`.

## Replacement Points

- Заменить `MockEquipmentCatalog` на каталог производителей, parsed datasheets или внутреннюю базу.
- Заменить pilot-координаты патрубков RGT-100 на координаты из проверенного заводского чертежа.
- Уточнить pilot-коллекторы STOUT DN32 по заводскому паспорту/чертежу: габариты корпуса, координаты выходов, применимость к котельной.
- Заменить `DemoInternalStandardsProfile` на реальные профили стандартов и правила компании/производителя.
- Заменить `SimpleOrthogonalPipeRouter` на роутер с обходом коллизий, ограничениями, уклонами, опорами и constructability-логикой.
- Заменить `MockPipeSizingService` на сервис гидравлического расчёта.
- Добавить реальные DXF, DWG, IFC и Revit exporters на местах существующих placeholders.
- Заменить mock AI adapters на реальные OpenAI/Codex/agentic workflows через существующие интерфейсы.

## Известные Ограничения

- v0 использует pilot/mock данные оборудования; RGT-100 основан на публичном паспорте, но остаётся `review_required`.
- Координаты патрубков RGT-100 условные; габариты и DN требуют проверки по актуальному заводскому чертежу.
- Коллекторы подачи/обратки заведены как pilot-модели на базе STOUT DN32 SDG-0016-005002 со статусом `review_required`; габариты корпуса и координаты выходов требуют сверки с паспортом/чертежом производителя.
- Совместимость точек подключения в v0 упрощённая.
- Редактор оборудования работает без database persistence.
- Нет реального каталога производителей.
- Нет PDF extraction и BIM import в runtime.
- Preview редактора карточки упрощённый.
- User overrides на уровне отдельного экземпляра оборудования пока не реализованы.
- Принципиальная схема и auto-connect пока поддерживают только базовый сценарий котёл -> подача/обратка.
- v0 использует условные placeholder-правила зон обслуживания.
- v0 строит трассы труб упрощённо и детерминированно.
- Принципиальная схема упрощена и не выбирает реальную гидравлическую схему автоматически.
- v0 не рассчитывает диаметры труб.
- v0 не покрывает полную MEP/BIM-логику, рабочую документацию и реальную проверку нормативов.
- Несколько коллекторов одного типа обрабатываются упрощённо: выбирается первый подходящий.
- Нет полноценного manual override editor для точек подключения.
- Нет BIM/Revit/IFC import и нет извлечения connector data из PDF в runtime.
- DXF export v1 пока без размеров, штампа, таблиц, блоков и hatch.
- Предпросмотр листа A3 является pilot drawing preview, а не рабочей документацией.
- Поворот поддержан для прямоугольников и точек подключения на 0, 90, 180 и 270 градусов, но семантика зон обслуживания остаётся упрощённой v0-моделью.
